1. Overview #

This blueprint describes a single, coherent stack for building mobile-first operations apps: scanner-heavy, role-gated workflows backed by a Frappe site, with a small Expo / React Native client and a single self-contained HTML developer doc.

The architectural choice that defines the whole approach: there is no installed Frappe app. All backend artefacts — DocTypes, Server Scripts, Custom Fields, Web Pages, Client Scripts, Custom HTML Blocks — live as JSON files in the repo and are pushed to a live Frappe site via MCP. Anyone with the repo and an MCP-connected site can rebuild the entire backend.

That choice cascades:

  • Backend changes are PRs, not bench commands.
  • Schema diffs read like normal code diffs.
  • Dev / staging / prod parity is "push the same JSON to a different URL".
  • You don't need to maintain a Python app.

5. Backend conventions: the mirror pattern #

Every backend artefact type that Frappe stores as a database record gets a JSON file in frappe/. Each file is a JSON array; each array element is an object whose top-level keys mirror the doc's fields:

[
  {
    "name": "<doc name>",
    "module": "<App Name>",
    "...": "..."
  }
]

Conventions

  • Sort order is irrelevant. Tools should sort consistently when writing.
  • Don't store auto-generated fields like creation, modified, owner, idx — they're noisy and recreate-able.
  • Store disabled: 0 explicitly for Server Scripts so dead scripts are visible.
  • Store the script field as a single JSON-escaped string with \n for newlines. Pretty-printing inflates noise.

Naming

  • API endpoints: <app>.<verb>_<noun> — e.g. harvest.scan_crate, clinic.book_appointment. The <app> prefix is a logical namespace, not a Python module path.
  • DocTypes: PascalCase, prefix with the app name when domain-specific (Harvest Team, Clinic Patient Visit). Generic doctypes (Vehicle, Employee) reuse Frappe / ERPNext defaults.
  • Custom Fields on stock ERPNext doctypes: prefix with custom_.
  • Roles: <App> <Role> — e.g. Loading Clerk, Receiver. Don't reuse stock Frappe role names.

6. Server Script house style #

Every API Server Script follows this shape. Copy and adapt.

try:
    # 1. Read body — fall back to form_dict so JSON & form-encoded both work
    data = {}
    try:
        data = frappe.request.get_json(silent=True) or {}
    except Exception:
        data = {}
    if not data:
        data = dict(frappe.form_dict or {})

    # 2. Pull and validate inputs
    foo = str(data.get('foo') or '').strip()
    qty = data.get('qty')
    try:
        qty = int(qty or 0)
    except Exception:
        qty = 0

    # 3. Role gating (if needed) — see §7 for why we query tabHas Role
    user = frappe.session.user
    role_rows = frappe.db.sql('SELECT role FROM `tabHas Role` WHERE parent = %s', (user,))
    roles = []
    for r in role_rows:
        if r and r[0]:
            roles.append(r[0])
    is_clerk = 'My Clerk Role' in roles

    # 4. Branch on validation — soft-fail with {error: "..."} on a 200
    if not is_clerk:
        frappe.response['message'] = {'error': 'Only Clerks can do this'}
    elif not foo:
        frappe.response['message'] = {'error': 'foo is required'}
    elif qty < 1:
        frappe.response['message'] = {'error': 'qty must be at least 1'}
    else:
        # 5. Do the work
        doc = frappe.new_doc('My Thing')
        doc.foo = foo
        doc.qty = qty
        doc.insert(ignore_permissions=True)
        frappe.db.commit()

        # 6. Build response payload
        frappe.response['message'] = {
            'name': doc.name,
            'foo': foo,
            'qty': qty,
        }
except Exception as e:
    frappe.db.rollback()
    frappe.log_error('<app>.<method>', str(e))
    frappe.response['message'] = {'error': str(e)}

Conventions

  • Always wrap in try / except — log via frappe.log_error(name, str(e)), return {error: str(e)}.
  • Never set frappe.response.http_status_code — the sandbox blocks setattr on frappe.response. Soft-fail on 200.
  • Prefer frappe.db.set_value for bulk updates — cheap, no doc-load overhead.
  • Use frappe.new_doc(...).insert(ignore_permissions=True) for new docs. Permissions are enforced by your role-gating, not Frappe's.
  • Use frappe.db.sql only for SELECT — DDL/DML raises "Query must be of SELECT or read-only WITH type."
  • Always frappe.db.commit() after a multi-doc transaction; on exception the outer rollback() undoes any work done before commit.

7. Frappe safe-exec sandbox #

Server Scripts run inside frappe.utils.safe_exec.safe_exec. The frappe object you see inside a script is not the real frappe Python module — it's a hand-curated NamespaceDict with a small allowlist of attributes. This single fact explains every restriction below.

This section is the canonical reference for what does and doesn't work. Every restriction here was discovered by hitting it in production. Save yourself the round-trip and read this carefully before writing your first script.

7.2 What does NOT exist on the frappe object in safe-exec

The injected frappe looks complete but is missing many attributes. These are the ones we have specifically tripped over:

  • frappe.rename_field — "module has no attribute 'rename_field'". Use the whitelisted REST endpoint instead (see workaround below).
  • frappe.get_attr — not exposed. Means you cannot dynamically resolve any whitelisted method through the sandbox.
  • frappe.get_roles — not exposed. You'd think this would work; it doesn't. Query tabHas Role directly.
  • frappe.model.*, frappe.permissions.*, etc. — submodule traversal silently resolves to None. Calling it gives "NoneType is not callable", which is the most confusing error in the sandbox.

WhyThe frappe object inside safe_exec is a frappe._dict NamespaceDict, not the real frappe Python module. It only knows about attributes the safe-exec author explicitly added. Anything else returns None through dict-lookup semantics.

7.3 What IS available

The reliable allowlist:

  • frappe.session.user — current user's name (typically email).
  • frappe.request.get_json(silent=True) — JSON body.
  • frappe.form_dict — form-encoded body fallback.
  • frappe.response['message'] — return payload.
  • frappe.db.sql — SELECT only.
  • frappe.db.set_value, frappe.db.get_value, frappe.db.exists, frappe.db.count, frappe.db.commit, frappe.db.rollback.
  • frappe.new_doc(...).insert(ignore_permissions=True).
  • frappe.get_doc(...) — loads a doc; further mutations respect normal perms unless you pass ignore_permissions to whatever you call.
  • frappe.rename_doc — auto-commits internally (the rename survives a later rollback()).
  • frappe.delete_doc, frappe.copy_doc.
  • frappe.utils.now_datetime(), frappe.utils.nowdate(), frappe.utils.add_days, frappe.utils.flt, frappe.utils.cint.
  • frappe.log_error(title, message).
  • lambda — usable in max(items.items(), key=lambda x: x[1]) patterns.
  • str(), int(), float(), list(), dict(), set(), tuple(), len(), range(), enumerate(), sorted(), min, max, sum, any, all — standard builtins are fine.

7.4 Common workarounds

Get the current user's roles

user = frappe.session.user
role_rows = frappe.db.sql('SELECT role FROM `tabHas Role` WHERE parent = %s', (user,))
roles = []
for r in role_rows:
    if r and r[0]:
        roles.append(r[0])
is_clerk = 'My Clerk Role' in roles

Rename a field

Impossible from inside a Server Script — frappe.rename_field isn't exposed and submodule traversal returns None. Call the whitelisted REST endpoint instead:

POST /api/method/frappe.model.rename_field.rename_field
  doctype=<dt>&old_fieldname=<old>&new_fieldname=<new>

Rename a DocType

frappe.rename_doc('DocType', 'Old Name', 'New Name', force=True, ignore_permissions=True)

Auto-commits internally — the rename survives a later rollback().

Add a column to a doctype

Don't ALTER TABLE — DDL is blocked. Push a Custom Field doc via MCP from outside the sandbox; Frappe runs the migration on save.

String formatting without .format()

# BAD: blocks
msg = "User {0} did {1}".format(user, action)

# GOOD: concatenation
msg = "User " + str(user) + " did " + str(action)

# ALSO GOOD: %-style
msg = "User %s did %s" % (user, action)

Build response payloads carefully

# Bad — sets http_status_code, blocked by sandbox
frappe.response.http_status_code = 500
frappe.response['message'] = {"error": "boom"}

# Good — soft-fail on 200; client wrapper promotes to throw
frappe.response['message'] = {"error": "boom"}

7.5 The "NoneType is not callable" trap

This is the single most confusing sandbox error. It happens when you do something like:

frappe.model.rename_field.rename_field(doctype, old, new)
# Error: 'NoneType' object is not callable

What's happening: the frappe NamespaceDict has no model attribute. Dict-lookup of an absent key returns None. Attribute access on None is intercepted to also return None (instead of raising). So frappe.model.rename_field.rename_field evaluates to None, and then you call it.

The fix: assume any frappe.<submodule> traversal will silently fail. Stick to top-level frappe attributes that you've verified work. When in doubt, write a one-line probe script first:

try:
    frappe.response['message'] = {"has_thing": frappe.thing is not None}
except Exception as e:
    frappe.response['message'] = {"error": str(e)}

7.6 Sandbox checklist (the pre-write review)

Before writing a new Server Script, scan this list:

  1. Am I importing anything? → No.
  2. Am I using .format() or f-strings? → Replace with + or %.
  3. Am I doing DDL/DML in frappe.db.sql? → Use set_value / insert() / delete_doc.
  4. Am I traversing frappe.<submodule>? → Stop. It returns None.
  5. Am I returning at the top level? → Use else: branches.
  6. Am I checking roles? → Query tabHas Role.
  7. Am I setting http_status_code? → Soft-fail on 200.
  8. Have I wrapped the whole body in try / except? → Yes, with frappe.log_error(name, str(e)) and {"error": str(e)}.

  "sort_order": "DESC",
  "fields": [
    {
      "fieldname": "foo",
      "label": "Foo",

      "fieldname": "status",

}

8.2 Custom Fields (preferred over editing core doctypes)

Stored in frappe/custom_fields.json. Use these to extend either your own DocTypes or stock ERPNext doctypes — they're easier to push via MCP than mutating the parent doctype's fields array.

{
  "dt": "My Thing",
  "fieldname": "extra_data",
  "label": "Extra Data",
  "fieldtype": "Data",
  "insert_after": "status"
}

NoteDirect edits to the parent doctype's child fields array via MCP often fail. Push a Custom Field doc instead. The parent doctype's fields array in doctypes.json stays as the source-of-truth view; the live data is the union of doctype fields + custom fields.

8.3 Property Setters

Stored in frappe/property_setters.json. Use to override a single property on a stock field.

{
  "doc_type": "Customer",
  "field_name": "tax_id",
  "property": "reqd",
  "value": "1",
  "property_type": "Check"
}

9. Client Scripts, Web Pages, Custom HTML Blocks #

9.1 Client Scripts

Stored in frappe/client_scripts.json. JavaScript that runs in the Frappe desk.

{
  "name": "My Thing — Quick Action",
  "dt": "My Thing",
  "view": "Form",
  "enabled": 1,
  "script": "frappe.ui.form.on('My Thing', { refresh(frm) { ... } });"
}

9.2 Web Pages

Stored as individual HTML files in frappe/web_pages/<route>.html. Pushed via the Web Page doctype's main_section_html field.

Pattern: a single self-contained file with all CSS and JS inline, calling frappe.call(...) for data. Auto-refresh every 30 s. No external libraries beyond what Frappe already loads.

9.3 Custom HTML Blocks

Stored in frappe/custom_html_blocks.json. The Frappe workspace navigation widgets.

{
  "name": "<App> Navigation",
  "private": 0,
  "script": "/* No JS needed */",
  "style": "...",
  "html": "..."
}

These render at the top of a workspace and act as a hub of links / shortcuts.

10. Deployment via MCP #

Pushing the repo state to a live Frappe site uses MCP tools — Claude calls them as create_document, update_document, delete_document. The pattern:

For each entry in frappe/<file>.json:
  if it doesn't exist on live → create_document
  else                        → update_document

Bootstrapping a new instance

  1. Create the Frappe site (Frappe Cloud, bench, etc.).
  2. Set up MCP access (an OAuth-connected Anthropic Frappe MCP server).
  3. From a clean repo, instruct Claude to push every doctype, then every server script, then custom fields, then property setters, then web pages, then custom HTML blocks. Stop on the first failure and report.

Order matters: doctypes before server scripts (scripts may reference the tables); custom fields after their parent doctypes exist; property setters last.

Updating a live site

After editing a JSON locally, instruct Claude to push only the changed entries (a git diff over the file is the simplest input).

Avoiding drift

A drift check is a one-shot script that for every entry in your local JSON, fetches the live record and compares the script body / fields. Easy to run before a release.

  );
}

13.2 The auth API (src/api/auth.ts)

import AsyncStorage from "@react-native-async-storage/async-storage";
import { getWorkingUrl } from "../utils/url";

function parseCookie(setCookieHeader: string | null, name: string): string | null {
  if (!setCookieHeader) return null;
  const re = new RegExp(`(?:^|[;,\\s])${name}=([^;,\\s]+)`);
  const match = re.exec(setCookieHeader);
  return match ? match[1] : null;
}

export async function login(email: string, password: string, urlInput: string) {
  const fullUrl = await getWorkingUrl(urlInput);
  if (!fullUrl) throw new Error(`Could not reach "${urlInput}".`);

  const body = new URLSearchParams();
  body.append("usr", email);
  body.append("pwd", password);

  const response = await fetch(`${fullUrl}/api/method/login`, {
    method: "POST",
    headers: { "Content-Type": "application/x-www-form-urlencoded" },
    body: body.toString(),
  });

  const text = await response.text();
  let parsed: any = text;
  try { parsed = JSON.parse(text); } catch { throw new Error("Invalid instance — please check the URL."); }
  if (!response.ok) {
    throw new Error(parsed?.message ?? parsed?.exc ?? `Login failed (${response.status})`);
  }

  const setCookie = response.headers.get("set-cookie");
  const sid = parseCookie(setCookie, "sid");
  const userId = parseCookie(setCookie, "user_id");
  if (!sid) throw new Error("Login succeeded but no session cookie returned");

  await AsyncStorage.setItem("instanceurl", fullUrl);
  await AsyncStorage.setItem("cookie", `sid=${sid}; user_id=${userId ?? ""}`);
  await AsyncStorage.setItem("user_email", email);

  return { fullUrl, sid, userId: userId ?? "" };
}

export async function logout() {
  await AsyncStorage.multiRemove(["instanceurl", "cookie", "user_email"]);
}

13.3 URL helper (src/utils/url.ts)

export async function getWorkingUrl(input: string): Promise<string | null> {
  const trimmed = input.trim().replace(/\/$/, "");
  if (!trimmed) return null;
  if (trimmed.startsWith("http://") || trimmed.startsWith("https://")) return trimmed;
  return `https://${trimmed}`;
}

If you want to probe multiple URL shapes — the user typed kaitet and you try https://kaitet.c.frappe.cloud and other guesses — extend this to call /api/method/ping on each candidate and return the first 200.

function extractFrappeError(err: AxiosError): string {
  const data = err.response?.data as any;
  if (data) {
    if (typeof data._server_messages === "string") {
      try {
        const parsed = JSON.parse(data._server_messages) as string[];
        const first = parsed[0];
        if (first) {
          try {
            const obj = JSON.parse(first);
            if (obj?.message) return obj.message;
          } catch { return first; }
        }
      } catch {}
    }
    if (typeof data.exception === "string" && data.exception) {
      const line = data.exception.split("\n").pop()?.trim();
      if (line) return line;
    }
    if (typeof data.message === "string" && data.message) return data.message;
    if (typeof data.message === "object" && data.message?.error) return data.message.error;
  }
  if (err.response?.status) return `Request failed (${err.response.status})`;
  return err.message || "Network error";
}

api.interceptors.response.use(
  (r) => {
    console.log("[api ←]", r.config.method?.toUpperCase(), r.config.url, r.status,
                "response:", r.data);

14.2 The typed call wrapper (src/services/api.ts)

import client from "../api/client";
import type { /* your types */ } from "../types";

async function call<T>(method: string, body: object = {}): Promise<T> {
  const res = await client.post<{ message: T }>(`/api/method/${method}`, body);
  const msg = res.data.message as unknown as { error?: string } | T;
  // Soft errors come back as 200 + {error: "..."} — promote them so callers
  // don't accidentally treat them as success.
  if (msg && typeof msg === "object" && "error" in msg && (msg as any).error) {
    throw new Error((msg as { error: string }).error);
  }
  return res.data.message;
}

16.9 Colors

A monochrome system reads as a single corporate product. Resist per-feature accents — distinguish sections by icons and layout, not colour. Keep COLORS to ~12 keys (see §12 for the canonical object).

16.10 Sounds

Three cue tones, preloaded once and replayed on demand. Source three short MP3s (≤500 ms each):

  • submit.mp3 — successful action (playSubmit() — call after a successful API write)
  • error.mp3 — failed action (playError() — call in the catch branch of any submit)

Files live in assets/audio/. Free sources: freesound.org, zapsplat.com, or generate with sfxr.

Sounds are not optional. They're the fastest feedback the user gets and let them keep their eyes on the physical task instead of the screen.

16.11 Iconography

Single icon library: @expo/vector-icons → MaterialCommunityIcons. Don't mix sets.

Icons are 18–24 px in headers, 16–20 px inline with text. Always paired with a text label except in dense list rows where context is unambiguous (e.g. close × on a removable chip).

16.12 Empty & loading states

Use the standard primitives, not bespoke ones:

  • Page-level loading: Screen's loading prop — centred spinner in the page accent colour.
  • Inline loading on a button: <Button loading />.
  • List empty state: an <Alert tone="info"> in the same Card, with one sentence telling the user why it's empty and what to do.
  • Page error: Screen's error + onRetry props.

No custom skeletons, no shimmer effects, no per-feature empty illustrations. The same emptiness reads the same everywhere.

// src/services/sounds.ts
import { AudioModule, createAudioPlayer, type AudioPlayer } from "expo-audio";

let scanPlayer: AudioPlayer | null = null;
let submitPlayer: AudioPlayer | null = null;
let errorPlayer: AudioPlayer | null = null;
let configured = false;

async function ensureConfigured() {
  if (configured) return;
  configured = true;
  try {
    await AudioModule.setAudioModeAsync({
      playsInSilentMode: true,
      shouldPlayInBackground: false,
    });
  } catch {}
}

function ensurePlayers() {
  if (!scanPlayer)   scanPlayer   = createAudioPlayer(require("../../assets/audio/beep.mp3"));
  if (!submitPlayer) submitPlayer = createAudioPlayer(require("../../assets/audio/submit.mp3"));
  if (!errorPlayer)  errorPlayer  = createAudioPlayer(require("../../assets/audio/error.mp3"));
}

function fireAndForget(p: AudioPlayer | null) {
  if (!p) return;
  try { p.seekTo(0); p.play(); } catch {}
}

export async function playBeep()   { await ensureConfigured(); ensurePlayers(); fireAndForget(scanPlayer); }
export async function playSubmit() { await ensureConfigured(); ensurePlayers(); fireAndForget(submitPlayer); }
export async function playError()  { await ensureConfigured(); ensurePlayers(); fireAndForget(errorPlayer); }

GotchaplaysInSilentMode: true is required for iOS to play with the ringer off — ops users almost always have their phones silenced.

19. Polling pattern #

The default data refresh strategy. Every screen that displays server state polls every POLL_INTERVAL_MS (default 30 s). Pull-to-refresh is wired through the Screen component's onRefresh prop.

const fetchState = useCallback(async () => {
  try {
    const data = await api.getThings();
    setThings(data);
    setError(null);
  } catch (e) {
    setError(e instanceof Error ? e.message : String(e));
  } finally {
    setLoading(false);
  }
}, []);

useEffect(() => {
  fetchState();
  const id = setInterval(fetchState, POLL_INTERVAL_MS);
  return () => clearInterval(id);
}, [fetchState]);

Don't introduce websockets / real-time subscriptions until you have a real reason. Polling is dumb, debuggable, cheap to rate-limit, and survives transient network failures.

If a screen has multiple data sources, batch them in one fetchState:

const fetchState = useCallback(async () => {
  const [a, b, c] = await Promise.all([
    api.getA(), api.getB(), api.getC(),
  ]);
  setA(a); setB(b); setC(c);
}, []);

20. OTA updates with EAS #

Once the app is shipped, ship JS-only changes via OTA. No Play Store rollout for every fix.

Setup (one-time)

npx expo install expo-updates

Add to app.json:

"runtimeVersion": { "policy": "appVersion" },
"updates": { "url": "https://u.expo.dev/<your-eas-project-id>" }

Add channel to each EAS build profile in eas.json:

"preview":    { "distribution": "internal", "channel": "preview" },
"production": { "autoIncrement": true, "channel": "production" }

Build one new APK to embed the OTA runtime:

eas build -p android --profile production

After every code change

eas update --branch production --message "Fix the thing"

The app pulls the update on next launch and applies it silently.

What OTA can NOT update

  • Native dependencies (new packages with native code).
  • New Android/iOS permissions.
  • SDK upgrades that require rebuilding native code.
  • Anything in app.json that affects native config.

For those, build a new APK.

23. Anti-patterns and lessons #

These are the things we did wrong at first and corrected. Skip them entirely.

23.1 Don't keep state offline when multiple users need it

If two phones are participating in the same workflow, the state belongs on the server. AsyncStorage drafts work for single-user pages but break the moment you need multi-user collaboration.

23.2 Always re-check roles on the server

UI hide is not security. Every clerk-only / receiver-only / admin-only action gets a server-side '<Role>' not in roles → error check. The UI flag is for UX only.

23.3 Idempotent endpoints over "already done?" queries

If an action is naturally idempotent (e.g. status flip with WHERE status = X filter), prefer the simple "click again, do nothing" model over tracking "has this been done?" state. Less schema, less code, less drift.

23.4 Single source of truth for each artefact

Don't have a Client Script and a mobile button do the same thing. Pick one (the mobile button) and delete the other. Two paths means you'll always wonder which the user used.

23.5 Server Scripts as the API surface — not RPC

Resist treating Server Scripts as "remote function calls that match what the UI happens to need." Each script is a resource verb: create_thing, release_thing, list_open_things. If the UI grows new needs, add a new script — don't bolt parameters onto an existing one until it's bloated.

23.6 No bench migrate dance

If you find yourself running bench migrate to sync schema, you're working too hard. Push schema changes via MCP Custom Field / DocType updates. The whole point of this stack is the JSON files are the schema.

23.7 Don't over-design the API client

The call<T> wrapper is 6 lines. Resist adding caching, retry, batching, request deduplication, optimistic updates. If a screen needs that, do it in the screen. The wrapper stays dumb.

23.8 Separate scripts dir for one-shots

Anything that's "run this once to migrate something" goes in scripts/, not in the main code. This keeps the active codebase's intent clear.

23.9 The single-file docs scale

Don't split the developer doc into 20 .md files. One big docs/index.html with sticky nav and search reads better, ships easier, and the search makes "where was that thing about X" trivial.

23.10 Soft errors > HTTP errors

Most of the things you'd want to return 4xx for (validation, business-rule violations, role rejections) ride happily on a 200 with {error: "..."}. The call<T> wrapper promotes them to thrown errors anyway, so the UI sees the same flow. Use real HTTP errors only for actual transport failures.

23.11 Polling beats websockets until it doesn't

Until you have a demonstrated user complaint about staleness, 30 s polling is the right answer. Keep it. The day someone says "I need this updated every second," that's when websockets earn their complexity.

23.12 Resist per-feature state machines

Most lifecycle problems collapse into a single status field on the canonical doctype with a small enum of values. Don't model state as separate doctypes ("Crate Allocation Open", "Crate Allocation Closed") — just status: Open | Closed and gate logic on it.

23.13 Probe the sandbox before assuming

When you're about to write a Server Script that uses something you haven't used before (a method on frappe.utils, a date format, a builtin), write a 3-line probe script first that just tries it and returns success/failure. Cheaper than 20 minutes of debugging "why is this returning None?".
